01 Positioning

项目定位

Synapse 是面向 AI 学习者的结构化学习社区。核心不是普通笔记,而是把数学推导、可运行代码、论文引用放进同一套 Note / Block / Link 语义模型里,让个人学习轨迹可以被检索、复习、公开、评论、引用和派生。

当前原型包含的主要界面

A1Marketing Landing
Hero、三类 Block、知识图谱演示、社区示例、工作流、Pricing、FAQ。
A2Workspace
个人学习首页、Roadmap、当前任务、最近 Note、复习提醒。
A3Studio
新建内容流程,包含标题、slug、标签、阶段、可见性、资源。
A4Note Editor
PaperBlock、MathBlock、CodeBlock、跨块 Link、自动保存、历史版本、发布。
A5Explore
Trending、Latest、Following、Tags、知识图谱、热门 Track。
A6Profile
用户主页、公开 Tracks / Notes / Papers / Graph、热力图、关注。
A7Review
SM-2 风格复习卡片、Deck、到期队列、准确率。
A8Auth
注册 / 登录、OAuth 入口、密码强度、产品价值说明。
A9Admin
后台管理、举报、注册闸门、审核策略、审计日志。
02 Domain Model

核心业务模型

2.1　领域实体

实体用途
User用户账号、角色、头像、bio、社交链接、可见性设置。
Track一条主题学习轨道,由多个 Note 组成。
Note最小可分享学习单元,由多个 Block 组成。
Block原子内容块,核心类型为 MathBlock、CodeBlock、PaperBlock。
LinkBlock 之间的一等公民关系,支持 implements、derives_from、cites、contradicts、extends、related。
ReviewCard从 Block / Note 派生的复习卡,用于 SM-2 复习。
Comment / Discussion评论和树状讨论。
Follow / Like / Bookmark社区互动和 Feed 来源。
Report / ModerationAction举报、审核、管理员审计。
Tag / Paper主题标签和规范化论文元数据。

2.2　产品主链路

  1. 用户注册并进入 Workspace。
  2. 用户创建 Track,按 Roadmap 推进学习。
  3. 用户在 Studio / Editor 创建 Note。
  4. Note 内插入 MathBlock、CodeBlock、PaperBlock。
  5. 用户在 Block 或 Math step 之间建立 Link。
  6. 保存草稿,发布为 private / unlisted / public。
  7. public Note 触发 embedding worker,进入推荐、搜索和图谱。
  8. 其他用户在 Explore / Profile / Graph 中浏览、关注、评论、引用。
  9. 用户把关键 Block 派生为 ReviewCard,按 SM-2 复习。
  10. 管理员处理举报、封禁、下架内容、调整注册策略。

05 Backend Stack

后端技术选型

5.1　基础栈

分类选择理由
语言Go 1.22+适合 API、后台任务、并发和长期维护。
Web 框架chi轻量,贴近标准库,中间件组合清晰。
数据访问sqlc + pgx类型安全 SQL,避免 ORM 黑盒。
迁移golang-migrate简单稳定,适合 CI 和 Docker。
数据库PostgreSQL 16主业务数据、全文 / 模糊搜索、事务一致性。
向量检索pgvectorNote / Block embedding 推荐和相似搜索。
模糊搜索pg_trgm搜索 Note、Paper、User、Tag。
缓存 / 限流Redis 7 + go-redis/v9token、限流、热点缓存、SSE 状态。
任务队列asynqembedding、邮件、PDF 元数据、举报通知。
认证JWT access + refresh cookie + Redis 吊销用户体验和后台管控平衡。
密码argon2id注册登录安全基线。
对象存储MinIO 本地 + S3 / R2 生产PDF、头像、图片。
日志zerolog结构化日志、低开销。
指标 / 追踪Prometheus + OpenTelemetry 接入点线上问题定位。
API 文档OpenAPI 3.1前后端协作、自动生成客户端类型。

5.3　API 风格

  • 主体使用 REST。
  • 通知和评论推送使用 SSE。
  • 错误码统一按 code / message / details / requestId 返回。
  • 前端所有 mutating 请求带 CSRF header。
  • 后台接口统一加 /admin 前缀并要求 role in (moderator, admin)。

07 Roadmap

开发顺序

共 10 个阶段。每个阶段都包含目标、交付与验收三段;点击阶段标题展开或收起明细。

目标　所有人能一条命令跑起前端、API、Postgres、Redis、MinIO。

交付 / Deliverables

  • 整理当前 Next.js 结构,补 .env.example。
  • 新增 services/api Go 服务,提供 /healthz。
  • 新增 Docker Compose:Postgres 16 + pgvector、Redis、MinIO。
  • 新增 Makefile 或 pnpm scripts:dev、api:dev、db:migrate、ci。
  • CI 先跑 pnpm typecheck、pnpm lint、go test ./...。

验收 / Acceptance

  • pnpm dev 可打开落地页。
  • go run ./services/api/cmd/server 可返回 /healthz。
  • docker compose up 可启动依赖。
目标　用户能注册、登录、刷新会话、访问个人身份。

交付 / Deliverables

  • users 迁移和 sqlc queries。
  • 注册、登录、刷新、退出、/auth/me。
  • argon2id、access token、refresh cookie、Redis 吊销。
  • 前端 Auth 页面接真实 API。
  • 基础 Route Guard。

验收 / Acceptance

  • 注册后可登录。
  • 刷新页面仍保持会话。
  • logout 后 refresh 失败。
目标　编辑器暂时可以先用简化 UI,但数据模型必须真实。

交付 / Deliverables

  • tracks、notes、blocks、block_links、papers 迁移。
  • Track CRUD、Note CRUD、Block 物化保存。
  • Note content 使用结构化 JSON,保存时同步 plaintext。
  • 前端 Workspace / Studio / Note Editor 读取真实 Track / Note。
  • Note visibility:private / unlisted / public。

验收 / Acceptance

  • 创建 Track。
  • 创建 Note 并保存三类 Block。
  • 刷新后数据完整恢复。
目标　实现产品差异化的 Math / Code / Paper / Link。

交付 / Deliverables

  • TipTap 编辑器基础。
  • MathBlock:CodeMirror LaTeX 输入、KaTeX 预览、step anchor。
  • CodeBlock:CodeMirror、多语言、复制代码、复制依赖。
  • PaperBlock:arXiv / DOI 导入、quote、anchor。
  • Link 创建弹窗:搜索 Block / Note、选择 relation type、写注释。
  • 自动保存:1.5s 防抖、失败提示、历史版本快照。

验收 / Acceptance

  • 一个 Note 中包含 PaperBlock、MathBlock、CodeBlock。
  • Math step 可以 link 到 CodeBlock。
  • hover link 显示对端预览。
  • 发布前后 Link 不丢失。
目标　公开内容可以被发现、浏览、关注和引用。

交付 / Deliverables

08 Principles

推荐开发原则

  • 先打通真实数据闭环,再精修视觉细节。
  • Note / Block / Link 是最高优先级,不要先把社区功能做满。
  • 每个 Phase 结束必须有可演示链路。
  • 后端先写迁移和 service 单测,再接 handler。
  • 前端所有 API schema 用 zod 或生成类型约束。
  • 编辑器复杂功能必须拆小:Block 渲染、Block 存储、Link 搜索、自动保存、历史版本分开做。
  • 所有外部能力,包括 embedding、邮件、OAuth、对象存储,都必须通过环境变量配置。
  • 任何 mock 都要可替换,并标明接口边界。

10 Open Questions

规格缺口清单

当前文档作为工程开发指南已完整,但作为产品规格仍有以下待补齐项。开发各 Phase 前应先关闭对应缺口, 避免前后端并行时返工。

高影响开发能否落地 中影响产品体验一致性 低易忽略但需提前定
高优先级 / Blocking
G1高

编辑器 Block / Link 规格

产品核心差异点但规格最薄。需补充:三类 Block 的 data jsonb 字段结构; Link 6 种关系的可连接 Block 类型约束与方向规则;Math step anchor 的定位机制; 历史版本的快照频率、保留期与回滚规则。
G2高

API 端点契约清单

已定义统一响应包络,但缺端点清单(路由、方法、入参、鉴权要求)。 Phase 1 起前后端并行,需要可对照的接口列表或 OpenAPI 草稿。
G3高

角色权限矩阵

仅有角色名(user / moderator / admin)。需一张权限对照表,覆盖三种角色对 Note 三种 visibility 的可见、评论、派生、编辑权限。
G4高

字段级数据模型

实体仅停在用途级。需补充枚举值全集(status、rel_type、report 状态等)、 关键字段类型与约束、外键级联策略。
中优先级 / Consistency
G5中

页面状态规范

每屏只有功能罗列,缺空状态、加载态、错误态、首次 onboarding 的设计与文案。
G6中

通知类型枚举

Notification 表已规划,但通知种类未列举(评论、关注、被引用、举报结果等)。
G7中

搜索排序权重

Trending 已有公式,但 keyword + tag + vector 的 hybrid 融合权重与排序规则未定义。
G8中

审核阈值默认值

敏感词表、自动隐藏阈值、新用户冷却规则均称"可配置"但无默认值,需给出初始基线。
G9中

SM-2 算法参数

称"简化算法",但 ease / interval 初始值与 Again / Hard / Good / Easy 的更新公式未给出。
G10中

Embedding 配置

embedding 模型、向量维度、长文 chunking 策略未定,直接影响迁移中 embedding 列定义。
易忽略但重要 / Easy to Miss
G11低

内容许可协议

Note 可公开、引用、派生,需明确内容授权方式(平台条款 / CC 协议),否则存在法律风险。
G12低

限流具体数值

分级限流只定义了层级,缺登录 / 发帖 / 评论各端点的具体阈值。
G13低

可观测性业务指标

已选 Prometheus,但需补充要埋点的业务指标与事件清单(注册转化、发布量、复习留存等)。
